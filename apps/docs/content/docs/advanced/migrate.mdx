Already have a docs site somewhere else? Migrating to Blume is an **agent-driven** task. Instead of a one-size-fits-all codemod, Blume ships the `blume-migrate` **skill** — a detailed playbook that teaches a coding agent (Claude Code, Codex) everything it needs to convert your repo _idiomatically_: translating the config, restructuring content into Blume's filesystem-derived navigation, rewriting components and callouts, and converting icons to Lucide. Docs repos differ too much for a reliable codemod; an agent handles the per-repo variance a codemod can't.

## Install the skill

The skill lives on GitHub, so you can install it **before** adding Blume to your project — you don't need a `blume` dependency first:

```bash
npx skills use haydenbleasel/blume@blume-migrate
```

Then, in your docs repo, run the skill in Claude Code:

```
/blume-migrate
```

Using another agent? Point it at `skills/blume-migrate/SKILL.md` (also bundled in the package at `node_modules/blume/skills/blume-migrate/` once Blume is installed) and ask it to migrate the repo.

:::warning
A migration **rewrites files in place** — pages, config, navigation, and assets. Commit (or stash) your work first so you can review the diff and roll back cleanly.
:::

## What the agent does

The skill detects your source framework (Mintlify, Docusaurus, Fumadocs, Nextra, Starlight, or any docs tool) and works a consistent loop:

<Steps>
  <Step title="Translate the config">
    The framework's config — `docs.json`, `docusaurus.config.*`, a `starlight({...})` block, and so on — becomes a `blume.config.ts`, mapping only what the source declares and relying on Blume defaults everywhere else.
  </Step>
  <Step title="Restructure content">
    Navigation moves from config into the **file tree** — folders become groups, numeric prefixes set order, `(group)/` folders group without a URL segment, and a typed [`meta.ts`](/docs/content/meta) refines a folder only where the filesystem can't. Source nav files (`_meta.*`, `meta.json`, `sidebars.js`) are translated across.
  </Step>
  <Step title="Rewrite pages">
    Each page is rewritten to idiomatic Blume markup: framework callouts become [`:::` directives](/docs/content/syntax), components are renamed or converted, snippets and includes are inlined, and frontmatter is mapped to Blume's strict [page schema](/docs/reference/frontmatter).
  </Step>
  <Step title="Convert icons to Lucide">
    Blume is [Lucide-only](/docs/content/components#icon). The agent maps each source icon (FontAwesome for Mintlify, Starlight's set, …) to its closest Lucide name, and reports any it can't map.
  </Step>
  <Step title="Adopt package.json and verify">
    The `dev`/`build`/`start` scripts are repointed at the `blume` CLI, old framework deps removed, and `blume` added. The agent then runs `blume build` to validate links, anchors, and frontmatter, and reports everything it dropped or approximated.
  </Step>
</Steps>

## After migrating

The agent ends with a written summary of what was migrated, what was **dropped** (navbar CTAs, footers, custom theming, dynamic redirects, unmappable icons, unsupported components), and suggested follow-ups. Then:

<Steps>
  <Step title="Review the config and diff">
    Open `blume.config.ts` and skim the diff. The agent maps conservatively — check the navigation, theme, and anything flagged in the summary.
  </Step>
  <Step title="Run the dev server">
    Serve the migrated site. Blume [validates your content](/docs/reference/cli#validating-links) as it serves it, so broken links, missing anchors, and schema errors surface immediately.

    ```bash
    blume dev
    ```

  </Step>
  <Step title="Customize further if needed">
    For deep customization the migration doesn't cover, [`blume add`](/docs/reference/cli) vendors a component into your project, and [`blume eject`](/docs/advanced/custom-pages) hands you a standalone Astro project.
  </Step>
</Steps>
